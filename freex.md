# 2. Почему появился FreeX

Андрей долго пользовался почти только ChatGPT.

Другие AI для него практически не существовали.

Позже появились реальные проблемы:

- длинные проекты теряли контекст;
- историю приходилось объяснять заново;
- решения расходились между чатами;
- важные мысли сохранялись вручную;
- ChatGPT иногда менял уже принятые решения;
- Obsidian требовал много ручного обслуживания;
- плагины Obsidian работали нестабильно;
- Notion помогал организовывать данные, но оставался чужой системой;
- Claude на телефоне плохо воспринимает очень длинные файлы и сообщения;
- переносить контекст между чатами, Notion, GitHub и агентами вручную стало слишком тяжело.

FreeX появился не как технология в поиске задачи.

Он появился из конкретной личной боли:

> Продолжать мысли, проекты и решения без постоянного начала с нуля.

# 3. Честное определение FreeX сегодня

FreeX сегодня — личная рабочая система Андрея для продолжения мыслей и проектов, объединяющая веб-приложение, заметки, память, нескольких AI-агентов, GitHub, документацию и рабочий компьютер.

FreeX пока не является готовым массовым Personal AI OS.

Это развивающийся личный инструмент, который позже может стать продуктом для других людей.

Главная рабочая формула:

> FreeX помогает продолжить мысль, решение или проект с того места, где человек остановился.

В разные моменты использовались метафоры:

- цифровой экзоскелет;
- Personal AI OS;
- продолжающаяся комната;
- личный продолжатель мышления.

Ни одна метафора пока не считается окончательной.

Важнее свойства:

- память;
- продолжение;

- основной точкой входа стал `js/app.js`;

- улучшена защита заметок и черновиков;

- tracked-секреты в изменениях не обнаружены;
- FreeX 8.1 проверен на телефоне и работает.

# 7. Система продолжения работы

Из-за обрывов Claude Code и ограничений AI были созданы:

- `PROJECT_STATUS.md`;
- `AI_RESUME.md`;
- `START_PROMPT.md`;
- `CHANGELOG.md`;
- команда `ок`.

В текущем рабочем процессе `ок` означает:

> Продолжить работу с точки обрыва, используя сохранённое состояние.

Это родилось как инженерный протокол, но стало важной продуктовой идеей.

Пользователь FreeX тоже постоянно прерывается.

Поэтому FreeX должен уметь восстанавливать:

- цель проекта;
- текущее состояние;
- уже принятые решения;
- открытые вопросы;
- незавершённые действия;
- следующий шаг;
- важные источники;
- причины прошлых решений.

Команда `ок` — рабочий механизм разработки, но не обязательно финальный интерфейс для пользователя.

# 9. Сильные идеи отдельных агентов

## DeepSeek

- второй мозг — процесс, а не склад;
- FreeX должен быть внимательнее, а не умнее;
- session memory / project memory / personal memory;
- честный экран ошибок;

- один идеальный сценарий важнее множества функций;
- память должна быть прозрачной и управляемой человеком.

## Qwen

- practical strategist и guardian простоты;
- сначала полезные связи, потом граф;
- один итог + скрытая работа Conference;
- память должна показываться простыми категориями;
- главный фильтр:

> Это помогает пользователю думать или только показывает, что мы можем?

## Kimi

- комната для мышления;
- продолжающаяся комната;
- ежедневный ритм;
- voice-first как важное направление;
- `Продолжаем` как ключевая способность;
- восстановление контекста должно стать продуктовой функцией.

## Grok

- Resume сам по себе не является уникальным moat;
- local-first без sync может ограничивать рост;
- Conference может превратиться в театр ролей;
- Personal AI OS — слишком широкое обещание;
- честная первая аудитория — Андрей и похожие независимые специалисты;
- лучше сильный узкий продукт, чем средний продукт для всех.

## Gemini

- цифровой экзоскелет;
- контекстное эхо;
- три уровня памяти;
- FreeX должен быть асимметрично тихим;
- опасность: пользователь обслуживает систему вместо работы.

## Perplexity

- память должна быть explainable;
- provenance важен;
- память не равна истине;
- факт, идея, гипотеза, решение и источник должны различаться.

## Copilot

- Project должен хранить цель, статус, решения, вопросы и следующий шаг;
- прозрачная память должна выглядеть как простой список;
- дневник и личные записи не должны автоматически анализироваться;
- Conference должна давать единый итог.

## NotebookLM

- FreeX как когнитивный экзоскелет;
- агенты должны иметь компетенции, а не только названия моделей;
- источник решения должен быть виден;
- человек должен сохранять контроль.

## SkyClaw

- гибрид direct providers и fallback;
- безопасный parsing ответов;
- ES Modules без тяжёлого сборщика;
- граф и wiki позже;
- безопасность ключей и performance важны.

## Atlas

- активная когнитивная поддержка;
- digest mode;
- защита от перегруза;
- automated testing до публичной версии.
