1. Общая философия

Для многолетней разработки крупного проекта правильный баланс:

1. Воспроизводимость среды — самое важное. Если через год компьютер сломается, ты должен восстановить всё за час, а не за неделю. Это определяет всё остальное.
2. Стабильность — важнее производительности. Компьютер, который работает без сбоев, экономит больше времени, чем компьютер, который на 10% быстрее, но иногда подвисает.
3. Простота обслуживания — система должна быть понятной. Если что-то сломалось, ты должен знать, как это исправить, не гугля три часа.
4. Безопасность — важна, но не должна мешать работе. Разработчик не может работать в системе, где каждый шаг требует подтверждения.
5. Производительность — на последнем месте. Современное железо достаточно быстрое. Проблемы производительности почти всегда вызваны неправильными настройками или раздутым софтом.

Мой баланс: воспроизводимость (40%), стабильность (25%), безопасность (15%), простота (15%), производительность (5%).

Что оставить несмотря ни на что

· BITS — используется для фоновых обновлений
· Hyper-V — если используешь Docker/WSL, оставь

5. Автозагрузка

Идеальная автозагрузка разработчика:

Автоматически:

· Windows Terminal (пустой, без сессий)
· VS Code (если есть незакрытые проекты)

Вручную:

· Docker (запускать только когда нужен)
· Браузер (запускать когда нужно)
· Git GUI (если используешь)
· Obsidian (когда нужен для заметок)

Правило: если программа не нужна сразу после загрузки — не добавляй её в автозагрузку.

6. Конфиденциальность

Действительно стоит изменить:

1. Отключить телеметрию  — через gpedit.msc или Chris Titus tool 
2. Отключить Activity History 
3. Отключить Location Tracking 
4. Отключить Consumer Features 

Не дают ничего:

· Отключение индивидуальной рекламы — не влияет на производительность
· Изменение настроек диагностических данных — если ты не в ЕС, эффекта почти нет

Могут неожиданно повлиять:

· Отключение Windows Update — делает систему небезопасной
· Отключение Defender — риск заражения
· Отключение Firewall — риск атаки

8. Производительность

Реальные источники потери производительности

1. Microsoft PC Manager — в Windows 11 24H2 эта служба может серьёзно тормозить компиляцию, неправильно распределяя нагрузку на ядра CPU 
2. Фоновые приложения — они потребляют память
3. Раздутый реестр — после установки/удаления ПО
4. Фрагментированные файлы — особенно на HDD (на SSD не так важно)

Что действительно ускоряет

1. GPU Hardware Acceleration — включить 
2. Отключение Game Bar — через реестр или политики 
3. Настройка питания — установить "High Performance" или "Ultimate Performance"
4. Отключение визуальных эффектов — через System Properties → Performance → Adjust for best performance
5. Очистка автозагрузки

Мифы

· "Отключение 20% зарезервированной пропускной способности" — не существует 
· "Очистка реестра ускоряет" — обычно нет
· "Дефрагментация SSD" — не нужно, даже вредно

Если памяти 8 ГБ

1. Отключить визуальные эффекты
2. Отключить SysMain
3. Закрывать браузер когда не нужен
4. Использовать легковесный редактор (VS Code, а не Visual Studio)
5. Отключить фоновые приложения

11. Структура проекта

```
C:\
├── Dev\
│   ├── freex\           # Основной проект
│   ├── freex-vscode\    # Расширение VS Code
│   └── experiments\     # Эксперименты
├── Tools\
│   ├── git\
│   ├── nodejs\
│   ├── python\
│   └── wsl\
├── Archive\
│   └── old-projects\    # Старые версии
├── Documents\
│   ├── knowledge\       # Знания, заметки
│   └── backups\         # Резервные копии
└── Temp\
    └── downloads\       # Временные файлы
```

Почему так:

· Dev — основная рабочая папка
· Tools — портабельные версии (не обязательны через %PATH%)
· Archive — старые проекты, которые не мешают
· Documents — знания, не код
· Temp — для временных файлов

13. План подготовки идеальной машины

1. Установка Windows
   · Скачать официальный ISO с Microsoft
   · Установить с отключённым интернетом (чтобы не создавать Microsoft аккаунт)
2. Базовые настройки
   · Включить Developer Mode 
   · Настроить питание на "High Performance"
   · Включить GPU Hardware Acceleration 
3. Обновления
   · Подключить интернет
   · Установить все обновления Windows (это может занять 30-60 минут)
   · Перезагрузиться
4. Установка инструментов
   · Git
   · GitHub CLI
   · Windows Terminal
   · VS Code (с расширениями)
   · Node.js + pnpm
   · Python
   · WSL2
   · Docker Desktop
5. Настройка приватности
   · Chris Titus Tool 
   · O&O ShutUp10++ (рекомендуемые настройки)
   · Отключить телеметрию 
   · Отключить Copilot (GPO) 
6. Настройка VS Code
   · Установить расширения (список выше)
   · Настроить синхронизацию настроек
7. Клонирование репозитория
   · git clone https://github.com/bestdreambot/freex.git
   · Установить зависимости
8. Создание резервной копии
   · Создать восстановительную точку
   · Записать список установленных программ

14. Типичные ошибки

1. Пропуск включения Developer Mode — потом не работают некоторые инструменты 
2. Удаление Edge WebView2 — перестают работать VS Code и другие приложения
3. Отключение Windows Update — система становится небезопасной
4. Установка "оптимизаторов" — они только захламляют систему
5. Хранение API-ключей в коде — большая ошибка
6. Не создание точки восстановления — потом некуда откатиться
7. Установка ненужных расширений VS Code — замедление редактора

15. Личный инженерный вердикт

Если бы я настраивал свою Windows для FreeX:

1. Я бы не делал сложной оптимизации. Сделал бы минимальные изменения: Developer Mode, GPU Hardware Acceleration, High Performance, отключил бы ненужные службы через Chris Titus Tool.
2. Я бы удалил приложения через PowerShell — Xbox, Teams, Clipchamp, Widgets, Weather, News, Phone Link, Copilot.
3. Я бы оставил Edge и WebView2 — они нужны для работы многих приложений.
4. Я бы сделал восстановительную точку после настройки, чтобы можно было быстро вернуться.
5. Я бы создал скрипт восстановления — один PowerShell-скрипт, который устанавливает всё необходимое. Если когда-нибудь придётся переустанавливать Windows, я просто запускаю скрипт и жду 20 минут.
6. Я бы не тратил время на микро-оптимизации. Каждая минута, потраченная на настройку, должна давать реальный эффект.

Главный принцип: Windows должна быть такой, чтобы я мог переустановить её за час и продолжить работать, будто ничего не случилось. Это важнее, чем 5% производительности.
