Отвечаю как **Инженер**.

Главный принцип: я бы не «выжигал» Windows скриптами. Для машины разработчика важнее **предсказуемость, обновления, восстановление и безопасность**, чем экономия условных 300 МБ RAM. Чистая Windows должна быть не минимальной, а контролируемой.

Microsoft подтверждает: на обычных редакциях Windows полностью отключить обязательные диагностические данные нельзя; цель — оставить только обязательную диагностику и выключить всё добровольное. См. [Microsoft Learn: Required Diagnostic Data](https://learn.microsoft.com/en-us/windows/privacy/required-diagnostic-events-fields-windows-11-24h2).

### 1. Что я бы удалил

Удалять — через **Параметры → Приложения → Установленные приложения**, а не агрессивным debloat-скриптом.

| Приложение | Решение | Зачем | Риск |
|---|---|---|---|
| Xbox, Xbox Game Bar, Game Bar plugins | Удалить | Не нужны для dev-машины, меньше фоновых задач и уведомлений | Потеря игровых функций |
| Clipchamp | Удалить | Если не монтируете видео | Нет |
| Microsoft Teams Personal | Удалить | Не путать с рабочим Teams, если он понадобится | Не будет личного Teams |
| Solitaire / рекламные игры / Spotify-триалы / OEM-предустановки | Удалить | Убирают рекламу и шум | Нет |
| Microsoft News / Start / Widgets | Удалить или отключить виджет | Меньше ленты, сетевой активности и отвлечений | Не будет новостной панели |
| Phone Link | Удалить, если не нужен | Полезен только при реальной интеграции с Android | Потеря синхронизации телефона |
| Maps / Get Help / Tips | Удалить по желанию | Не являются частью dev-потока | `Get Help` иногда удобен для диагностики |
| OneDrive | Не удалять сразу; отключить автозапуск | Пока не выбран план резервного копирования, лучше не ломать интеграцию | Удаление может создать путаницу с Desktop/Documents sync |

Лично я бы **не удалял системные Appx-пакеты через PowerShell массово**, особенно с `-AllUsers` и удалением provisioned-пакетов. Такие «оптимизации» нередко ломают Microsoft Store, Windows Update, вход в аккаунт, установку зависимостей и будущие обновления. Даже рекомендации по удалению bloatware подчёркивают необходимость осторожности. См. [PDQ: удаление bloatware](https://www.pdq.com/blog/how-to-remove-bloatware/) и [Win11Debloat](https://github.com/raphire/win11debloat).

### 2. Что обязательно оставить

- **Microsoft Defender + SmartScreen** — оставить включёнными. Это базовая защита для машины, где будут GitHub-токены, API-ключи, SSH-ключи и пакеты из npm.
- **Windows Update** — только режим «вручную выбрать время перезапуска», но не отключение.
- **Microsoft Store** — нужен для обновления некоторых штатных компонентов и удобной установки отдельных утилит.
- **App Installer** — даёт `winget`; для dev-машины это один из ключевых компонентов.
- **Windows Terminal** — основной терминал для Git, PowerShell, WSL и Node.js.
- **Notepad** и **Snipping Tool** — быстрые системные инструменты без тяжёлых альтернатив.
- **Калькулятор** — банально, но удобен.
- **Windows Security** — не удалять и не отключать уведомления полностью.
- **Recovery Environment / System Restore** — критично сохранить.
- **WebView2 Runtime** — многие приложения используют его; не удалять.
- **Microsoft Edge** — даже если основной браузер другой: нужен для некоторых системных страниц, WebView-компонентов и как резервный браузер.

### 3. Что часто удаляют по ошибке

Не удалял бы:

- **Microsoft Store / App Installer** — ломает удобный `winget`-поток.
- **Edge WebView2 Runtime** — может поломать VS Code-расширения и другие desktop-приложения.
- **Windows Update Medic Service**, **Update Orchestrator**, **BITS** — после этого обновления начинают зависать или перестают ставиться.
- **Defender**, **SmartScreen**, **Firewall** — «быстрее» не станет заметно, зато резко возрастёт риск.
- **VSS / Volume Shadow Copy** — нужен для точек восстановления и некоторых сценариев бэкапа.
- **Print Spooler** — если «принтера нет», его часто выключают; но тогда перестают работать печать в PDF и другие программы, которые используют печатный API.
- **Search Indexer** — лучше ограничить индексирование, а не отключать: поиск по проектам, файлам и настройкам станет хуже.
- **OneDrive до организации бэкапа** — сначала настроить альтернативу, потом решать.

### 4. Что можно отключить

Отключать только если функция действительно не используется. Перед этим — создать точку восстановления.

| Служба / функция | Лично рекомендую? | Польза | Последствие |
|---|---:|---|---|
| `RetailDemo` | Да | Убирает демонстрационный режим магазина | Нет для обычного ПК |
| `Fax` | Да | Не нужна почти никому | Не будет факса |
| `Remote Registry` | Да | Меньше поверхность атаки | Нельзя удалённо менять реестр |
| Xbox-службы | Да, если без игр | Меньше фонового шума | Не будут работать Xbox-функции |
| Downloaded Maps Manager | Да, если не нужны offline-карты | Меньше фоновых данных | Не будет offline Maps |
| Windows Error Reporting | Скорее нет | Минимум сетевой активности | Теряется полезная диагностика падений |
| Connected User Experiences and Telemetry | Осторожно | Меньше необязательной телеметрии | Некоторые настройки/диагностика могут вести себя нестабильно |
| Bluetooth Support Service | Только на стационарном ПК без Bluetooth | Меньше процессов | Не заработают гарнитура, телефон, мышь |
| Print Spooler | Нет по умолчанию | Небольшое уменьшение поверхности атаки | Проблемы с печатью/PDF |
| SysMain | Только если видите реальную нагрузку диска | Может уменьшить постоянные обращения к диску | На современных SSD обычно пользы от отключения мало |

Лично: отключил бы только `RetailDemo`, `Fax`, `Remote Registry` и Xbox-компоненты при отсутствии игр. Остальное — **не трогал бы без конкретной причины**.

### 7. Настройки, которые я изменил бы первыми

1. **Обновить Windows полностью**, включая optional driver updates — но драйвер видеокарты лучше брать с сайта производителя GPU.
2. Установить нормальное имя компьютера: например, `FREEX-DEV-01`.
3. Включить **BitLocker / Device Encryption**. Ключ восстановления сохранить вне компьютера.
4. Создать:
   - один обычный ежедневный аккаунт;
   - отдельный администраторский аккаунт для установки системного ПО.
5. Настроить Windows Update:
   - активные часы;
   - уведомление перед перезапуском;
   - не отключать обновления.
6. Показать расширения файлов в Проводнике.
7. Включить буфер обмена с историей — `Win + V`, если вам это удобно.
8. Настроить Windows Terminal как терминал по умолчанию.
9. В PowerShell включить исполнение только локальных скриптов по необходимости, а не `Unrestricted` глобально.
10. Создать первую точку восстановления до «чистки».
11. Настроить регулярный экспорт FreeX-данных и GitHub-бэкап проекта.

### 9. Типовые ошибки «идеально чистой Windows»

1. Запуск неизвестного debloat-скрипта с правами администратора.
2. Отключение Windows Update и Defender.
3. Удаление Store, App Installer, WebView2 и Edge.
4. Отключение 20–50 служб «по списку из YouTube».
5. Отсутствие точки восстановления и проверенного бэкапа.
6. Хранение API-ключей, GitHub PAT и SSH-ключей в `.env`, который потом коммитится.
7. Работа всегда из Administrator-аккаунта.
8. Установка десятков VS Code-расширений «на всякий случай».
9. Установка Node.js, Python и Git из случайных источников.
10. Попытка сделать «zero telemetry» ценой нестабильной ОС.

### 10. Как бы я настроил свою машину под FreeX

#### Первый запуск

1. Устанавливаю **Windows 11 Pro**, если есть выбор: удобнее политики, BitLocker и удалённое администрирование. Home тоже годится, но Pro практичнее для dev-машины.
2. Выбираю локальную учётную запись, если это возможно и соответствует вашему сценарию.
3. На экранах приватности выключаю всё необязательное.
4. После первого входа ставлю обновления и перезагружаюсь столько раз, сколько нужно.
5. Ставлю драйверы чипсета, Wi‑Fi/LAN, GPU и BIOS/firmware **только с сайта производителя компьютера или компонентов**.

#### Безопасность и восстановление

6. Включаю BitLocker / Device Encryption.
7. Сохраняю recovery key в безопасном независимом месте.
8. Проверяю Defender, Firewall и SmartScreen.
9. Создаю restore point: `Clean Windows — updated`.
10. Настраиваю резервную стратегию:
    - проект FreeX — GitHub;
    - экспорт заметок и чатов — регулярно;
    - важные документы — отдельное облако или внешний диск;
    - не считать GitHub заменой полного бэкапа.

#### Инструменты разработки

11. Устанавливаю через `winget`, где возможно:
    - Git;
    - VS Code;
    - Node.js LTS;
    - Windows Terminal;
    - браузер для разработки;
    - 7-Zip;
    - при необходимости Python;
    - при необходимости Docker Desktop;
    - при необходимости WSL2 + Ubuntu.

12. Для FreeX я бы установил:
    - Node.js LTS;
    - Git;
    - VS Code;
    - расширения VS Code: ESLint, Prettier, GitHub Pull Requests, Live Server или аналог, Markdown Preview;
    - Chrome/Edge DevTools для PWA, Service Worker и IndexedDB.

13. Настраиваю Git:
    - имя и email;
    - SSH-ключ с passphrase;
    - ключ добавляю в GitHub;
    - включаю двухфакторную аутентификацию GitHub;
    - PAT создаю с минимальными правами и коротким сроком, если он вообще нужен.

14. Клонирую `freex` в понятную структуру, например:

```text
C:\dev\freex
C:\dev\experiments
C:\dev\archive
```

#### Рабочие правила

15. Не храню реальные ключи в коде и не коммичу `.env`.
16. Добавляю `.env.example` без секретов.
17. Перед любым большим рефакторингом создаю ветку и проверяю локально.
18. Для FreeX настрою:
    - локальный запуск;
    - тестирование в mobile viewport;
    - DevTools → Application → IndexedDB;
    - DevTools → Application → Service Workers;
    - offline-проверку;
    - Lighthouse PWA audit.

19. После полной настройки создаю образ системы или хотя бы второй restore point: `FreeX Dev Base`.

### Мой итог

Я бы строил эту машину не как «облегчённую Windows», а как **надёжную личную инженерную станцию**:

- минимум лишних приложений;
- никаких агрессивных оптимизаторов;

Это даст больше скорости разработки и меньше неожиданных поломок, чем радикальная «деблоатизация».
