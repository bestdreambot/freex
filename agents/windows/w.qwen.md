## 1. Главная философия

**Приоритеты для идеальной Windows разработчика:**

```
1. Воспроизводимость среды > 2. Стабильность > 3. Простота восстановления
4. Безопасность > 5. Удобство ежедневной работы > 6. Производительность
```

**Почему именно так:**

🔹 **Воспроизводимость** — если ты можешь воссоздать среду за 2 часа на любом компьютере, ты свободен. Если среда «магическая» и держится на твоей памяти — ты в ловушке.

🔹 **Стабильность** — разработчик не должен тратить умственную энергию на «почему сегодня не работает». Предсказуемость > пиковая скорость.

🔹 **Простота восстановления** — диск умер, Windows переустановлена — ты должен вернуться к работе, а не к настройке.

🔹 **Безопасность** — API-ключи, SSH, исходный код — это активы. Потеря = катастрофа.

🔹 **Удобство** — важно, но вторично: неудобный инструмент можно терпеть, неработающий — нет.

🔹 **Производительность** — для веб-разработки (FreeX) 8 ГБ RAM и SSD достаточно. Оптимизация «ради 5%» часто ломает стабильность.

**Главный принцип:** «Настрой один раз, забудь, работай».

## 3. Компоненты Windows — три группы

### ✅ Обязательно оставить

- **Windows Defender** — реальная защита, минимальное влияние на производительность при правильной настройке
- **Windows Firewall** — базовая сетевая безопасность
- **Windows Update** — критичные обновления безопасности; можно настроить «паузу», но не отключать
- **Cryptographic Services** — требуется для SSH, TLS, подписи кода
- **Windows Installer** — установка .msi-пакетов
- **Edge WebView2 Runtime** — см. выше
- **WSL2** — если используешь Linux-инструменты (Docker, bash-скрипты)

### 🟡 Можно отключить при условиях

- **SmartScreen** — если доверяешь источникам софта; но оставляет защиту от фишинга в Edge
- **Search (индексация)** — если мало файлов или используешь альтернативный поиск (Everything); но отключение замедляет поиск в Проводнике
- **SysMain (Superfetch)** — на SSD эффект минимален; можно отключить, если видишь высокую дисковую активность
- **BITS** — фоновая загрузка обновлений; можно ограничить, но не отключать полностью

### 🔴 Никогда не трогать

- **WinRE (Windows Recovery Environment)** — единственный способ восстановить систему без переустановки
- **Windows Update (полное отключение)** — приводит к уязвимостям, поломкам совместимости
- **Plug and Play, RPC, DCOM** — ломают базовую функциональность системы

| Служба | Зачем нужна | Влияние на производительность | Стоит ли отключать | Риски |
|--------|-------------|------------------------------|-------------------|-------|

| **Windows Update** | Обновления системы | Фоновое, но может занимать диск/CPU во время установки | ❌ Нет | Уязвимости, поломки совместимости |
| **Defender Antimalware Service** | Сканирование в реальном времени | Низкое при исключениях для папок разработки | ⚠️ Настроить исключения, не отключать | Риск заражения, ложные срабатывания на скрипты |
| **Connected User Experiences** | Телеметрия | Минимальное | ✅ Можно отключить | Никаких (кроме потери телеметрии) |

**Принцип:** Отключать только то, что точно не нужно. Не гнаться за «+2% производительности» ценой стабильности.

## 6. Конфиденциальность — что действительно полезно

### ✅ Полезные настройки

- **Отключить рекламный ID** (`Настройки → Конфиденциальность → Общие`)
- **Запретить приложениям доступ к местоположению, камере, микрофону по умолчанию**
- **Отключить «Отправлять данные о вводе»** (телеметрия клавиатуры)
- **Ограничить диагностику до «Обязательных данных»**

### ⚠️ Мало что меняют

- «Отключить все разрешения сразу» — многие приложения перестанут работать, придётся включать обратно
- «Блокировать телеметрию через hosts» — ломает обновления, сложно поддерживать

### ❌ Могут вызвать проблемы

- Полное отключение телеметрии через сторонние скрипты — может сломать Windows Update, Store, активацию
- Удаление компонентов приватности через Debloat-скрипты — риск нестабильности

**Принцип:** «Минимальная достаточность» — отключи явный сбор данных, но не ломай системные зависимости.

## 7. Безопасность — хранение секретов

### SSH-ключи
- Хранить в `%USERPROFILE%\.ssh\`
- Права: только твой пользователь (`icacls`)
- Никогда не коммитить в Git
- Использовать `ssh-agent` для удобства

### GitHub / API-ключи
- **Не хранить в plain text** в `.env` на диске
- Использовать **VS Code SecretStorage** (для расширений)
- Для проектов: `.env` + `.gitignore` + шифрование через `git-crypt` или `sops` при необходимости
- Для локальной разработки: достаточно `.gitignore` + здравый смысл

### .env и секреты проекта
- `.env` — только локально, никогда в репозитории
- `.env.example` — шаблон без значений — в репозитории
- Для команд: использовать GitHub Secrets или аналог

### Частые ошибки
1. Закоммитить `.env` по ошибке → сразу отозвать ключи
2. Хранить ключи в коде («захардкодить») → утечка при публикации
3. Использовать один ключ для всех сред → компрометация = крах всего
4. Не ротировать ключи → долгосрочный риск

**Правило:** «Ключ, который ты не можешь отозвать за 5 минут — уже не твой».

## 8. Производительность — мифы и реальность

### ✅ Что действительно ускоряет

- **SSD вместо HDD** — главный прирост
- **16 ГБ RAM вместо 8 ГБ** — для Docker, нескольких вкладок, IDE
- **Отключение визуальных эффектов** (`Система → Дополнительные параметры → Быстродействие`) — если интерфейс «тормозит»
- **Исключения в Defender для папок разработки** — снижает нагрузку при компиляции
- **Настройка схемы электропитания на «Высокая производительность»** — для десктопа

### ❌ Мифы

- «Отключить службы = +30% скорости» — на современных CPU/SSD эффект <5%, риск нестабильности
- «Регулярная дефрагментация» — SSD не нужно, Windows делает это сама
- «Очистка реестра» — не даёт прироста, может сломать систему
- «Отключить индексацию поиска» — экономит немного CPU, но поиск становится бесполезным

### Если всего 8 ГБ RAM — где выигрыш

1. **Закрывай лишние вкладки браузера** — Chrome/Edge едят RAM
2. **Используй pnpm вместо npm** — меньше дублирования node_modules
3. **Отключи тяжелые расширения VS Code** — каждое +50-200 MB
4. **Не держи Docker запущенным, если не используешь** — 1-2 ГБ в фоне
5. **Настрой файл подкачки на SSD** — лучше медленный swap, чем краш

## 11. Организация файлов — архитектура

```
C:\
├── Users\<ты>\
│   ├── .ssh\              # SSH-ключи
│   ├── .config\           # Конфиги инструментов (git, gh, etc.)
│   ├── Documents\         # Личные документы
│   └── AppData\           # Системное, не трогать
│
├── Dev\                   # Среда разработки
│   ├── tools\             # Установленные вручную утилиты
│   └── env\               # Виртуальные окружения, если нужно
│
├── Projects\              # Исходный код
│   ├── freex\             # Основной проект
│   ├── experiments\       # Песочница, прототипы
│   └── archive\           # Завершённые проекты (read-only)
│
└── Archive\               # Личный архив, бэкапы, медиа
    ├── backups\           # Экспорты FreeX, конфигурации
    └── references\        # Документация, скриншоты, референсы
```

**Почему так:**
- `Dev\` — инструменты, которые можно переустановить
- `Projects\` — исходный код, под версионным контролем
- `Archive\` — данные, которые не в Git, но важны

**Бэкап-стратегия:**
- `Projects\` — GitHub (основной) + локальная копия
- `Archive\backups\` — внешний диск / облако (по выбору)
- `.ssh\`, `.config\` — экспорт в зашифрованный архив раз в месяц

## 13. Полный план подготовки — от чистого Windows до FreeX

### День 1: Базовая настройка

```
1. Обновить Windows до последней версии
2. Настроить учётную запись (локальная, не Microsoft, если не нужна синхронизация)

4. Установить драйверы (видео, сеть) с сайта производителя
5. Настроить схему электропитания: «Высокая производительность»
6. Отключить автозапуск лишнего (OneDrive, Teams, Xbox)
7. Включить отображение расширений файлов, скрытых файлов
8. Настроить Defender: добавить исключения для C:\Dev, C:\Projects
```

### День 2: Инструменты разработчика

```
9. Установить Git, GitHub CLI, Node.js LTS, pnpm
10. Установить VS Code + включить Settings Sync
11. Установить Windows Terminal, сделать PowerShell 7 по умолчанию
12. Настроить SSH: сгенерировать ключ, добавить в GitHub
13. Установить базовые расширения VS Code (GitLens, Prettier, etc.)
14. Протестировать: git clone, npm install, npm run dev

### День 3: FreeX-специфика

```
15. Создать структуру папок: C:\Projects\freex
16. Клонируй репозиторий FreeX
17. Установить зависимости, запустить локально
18. Настроить .env (ключи API) — только локально, в .gitignore
19. Протестировать: запрос к DeepSeek, сохранение заметки
20. Создать первый бэкап конфигурации (скриншоты, список пакетов)

### Финальная проверка

```
✅ VS Code открывает проект
✅ git push/pull работает
✅ npm run dev запускает FreeX
✅ Запрос к API модели возвращает ответ

✅ После перезагрузки — всё на месте
```

**Время:** 3 дня по 2-3 часа. Не торопиться.

## 14. Типичные ошибки разработчиков

1. **«Установлю всё на всякий случай»** → захламление, конфликты, медленная отладка
2. **Не настраиваю .gitignore сразу** → случайный коммит секретов
3. **Храню ключи в коде** → утечка при публикации
4. **Отключаю Defender / Update ради «производительности»** → уязвимости
5. **Не делаю бэкап конфигурации** → переустановка = день настройки заново
6. **Использую кастомные ISO (Tiny11 и т.п.)** → нестабильность, проблемы с драйверами
7. **Синхронизирую всё через OneDrive** → конфликты в node_modules, .git
8. **Не тестирую восстановление** → думаешь, что всё под контролем, пока не случится сбой

**Как избежать:** Чек-лист выше + правило «настраивай только то, что используешь».

## 15. Мой личный инженерный подход

Если бы это была моя основная машина на 3-5 лет:

### Сделаю обязательно

✅ Чистая Windows 11 с официального сайта  
✅ Локальная учётная запись (без привязки к Microsoft, если не нужна)  
✅ Точечная настройка приватности через O&O ShutUp10++  
✅ Исключения в Defender для папок разработки  
✅ Структура `C:\Projects\`, `C:\Dev\`, `C:\Archive\` с первого дня  
✅ Git + GitHub CLI + SSH-ключи — настроены до установки чего-либо ещё  
✅ VS Code Settings Sync — чтобы восстановить среду за 10 минут на любом ПК  
✅ Еженедельный экспорт конфигурации (список пакетов, скриншоты настроек)  

### Никогда не сделаю

❌ Не буду использовать кастомные ISO (Tiny11, AtlasOS) — риск безопасности  
❌ Не буду отключать Windows Update полностью — только пауза при необходимости  
❌ Не буду удалять системные компоненты через Debloat-скрипты — только ручное, с пониманием  
❌ Не буду хранить секреты в plain text без .gitignore  
❌ Не буду устанавливать «на всякий случай» — только по мере необходимости  

### Философия

> «Среда должна быть скучной. Если ты думаешь о системе — значит, она мешает работе».

FreeX — это про мышление, не про настройку Windows. Сделай среду предсказуемой, безопасной, восстанавливаемой — и забудь о ней.
