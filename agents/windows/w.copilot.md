1. Встроенные приложения Windows — что трогать, а что нет

Общий принцип: удаление стандартных приложений почти не даёт прироста производительности, но может ломать сценарии, обновления и интеграции. Для машины под FreeX лучше не превращать систему в «кастомный Frankenstein».

Я бы НЕ удалял, а просто отключал/игнорировал:

• Xbox, Movies & TV, Groove Music, News, Weather, Widgets:
Зачем существуют: потребительские сервисы.
Для разработчика: почти бесполезны.
Польза от удаления: минимальна, можно просто не запускать.
Риск: иногда цепляются за системные компоненты, обновления.
Решение для FreeX: оставить, но не использовать.
• Mail, Calendar, People:
Зачем: базовые PIM-приложения.
Для разработчика: можно заменить вебом.
Польза от удаления: почти нулевая.
Риск: ломать интеграции с системными уведомлениями.
Решение: не удалять, просто не настраивать.
• 3D Viewer, Paint 3D, Clipchamp:
Зачем: креативные инструменты.
Для FreeX: не критичны.
Польза от удаления: символическая.
Риск: минимальный, но смысла мало.
Решение: можно удалить, если очень хочется чистоты, но это косметика.


Важно: не трогать системные компоненты, даже если они выглядят как «лишние приложения».

2. Что никогда не стал бы удалять

Microsoft Store:

• Нужен для установки App Installer, Winget, обновлений VS Code, драйверов.
• Удаление ломает удобный канал установки и обновлений.


App Installer / Winget:

• Это твой пакетный менеджер.
• Идеален для быстрой установки Node, Git, VS Code, утилит.


Edge WebView2 Runtime:

• Многие приложения (включая VS Code, некоторые dev-инструменты) используют WebView2.
• Удаление ломает UI и встроенные веб-компоненты.


Defender / SmartScreen / Firewall:

• Это базовая защита.
• Отключение ради «ускорения» — плохая идея, особенно на машине с реальными ключами и репозиториями.


WinRE (Recovery Environment):

• Нужен для восстановления системы.
• На dev-машине, где ты экспериментируешь, это страховка.


Windows Update:

• Важно для безопасности, драйверов, совместимости.
• Можно настроить, но не отключать.

3. Службы Windows — что оставить, что не трогать

Оставить без изменений:

• Windows Update, Defender, Firewall, SmartScreen, Cryptographic Services, Task Scheduler, COM+ Event System, Windows Installer.
Они — основа нормальной работы системы и разработки.


Можно аккуратно отключить (через Services / Settings, не через реестр):

• Bluetooth Support, Fax, Print Spooler — если точно не используешь принтер/факс/BT.
• Remote Registry — обычно не нужен.
• Secondary Logon — если не используешь RunAs.


Популярные ошибочные советы:

• «Отключите Defender, будет быстрее» — риск безопасности выше, чем выигрыш.
• «Отключите Windows Update» — ломает совместимость и безопасность.
• «Отключите Telemetry полностью» — можно уменьшить, но радикальное отключение иногда ломает Store, лицензии, обновления.

4. Автозагрузка — идеальный минимум

Автостарт:

• VS Code (опционально): если ты всегда начинаешь день с него.
• GitHub Desktop / SourceTree (если используешь): можно вручную, не обязательно.
• Синхронизация (OneDrive/Dropbox/GitHub Codespaces) — если реально нужна.


Только вручную:

• Браузеры (Edge/Chrome/Firefox).
• Node-сервисы, локальные dev-серверы.
• Docker Desktop (если используешь — он тяжёлый, лучше запускать по необходимости).
• Любые мессенджеры — чтобы не отвлекали.


Принцип: автозагрузка — только то, без чего ты не начинаешь рабочий день. Всё остальное — по запросу.

5. VS Code — расширения

Сразу поставить:

• ESLint / Prettier: единый стиль кода.
• GitLens: глубокий Git-контекст, история, blame.
• Path Intellisense / Auto Import: удобство навигации по проекту.
• Bracket Pair Colorizer / Indent Rainbow (или встроенные аналоги): визуальная структура.
• Markdown All in One: для твоих .md (copilot.md, PROJECT_STATUS.md и т.д.).


Осторожно:

• Тяжёлые LSP-расширения для десятков языков, если ты ими не пользуешься.
• Много тем и икон-паков — они не вредят, но засоряют.
• Расширения, которые дублируют функциональность (несколько форматтеров, несколько Git-панелей).


Не ставить без необходимости:

• «Оптимизаторы» и «чистильщики» внутри VS Code.
• Сомнительные AI-плагины, которые требуют свои ключи и лезут в твой код без понятной пользы.

6. Node.js

Версия:

• LTS (например, 20.x) — стабильность важнее новизны.


Инструменты:

• nvm-windows или аналог — для управления версиями Node.
• npm / pnpm / yarn — выбери один основной (я бы взял pnpm для скорости и чистоты).
• TypeScript — даже если проект на JS, TS полезен для типизации и проверки.

7. Git — базовая настройка

Сразу:

• git config --global user.name "Andrey"
• git config --global user.email "..."
• git config --global core.autocrlf true (для Windows, чтобы не ломать переносы строк).
• git config --global pull.rebase false (если не хочешь сложных конфликтов).


До первого коммита:

• Настроить .gitignore (node_modules, .env, build-артефакты).
• Убедиться, что ключи и секреты не лежат в репозитории.
• Проверить git status — никаких случайных файлов.

8. Безопасность — ключи и секреты

GitHub / SSH:

• Использовать SSH-ключи, хранить их в %USERPROFILE%\.ssh.
• Не коммитить .ssh ни при каких условиях.
• Можно использовать Git Credential Manager для HTTPS, но SSH надёжнее.


API-ключи / .env:

• Хранить в .env.local, .env.development, не коммитить.
• В FreeX — ключи в localStorage/IndexedDB, не в коде.
• Никогда не писать ключи в README, примерах, скриншотах.


Частые ошибки:

• Коммит .env «случайно».
• Публикация ключей в гистах.
• Отправка скриншотов с видимыми токенами.

9. Производительность на 8 ГБ RAM

Где реально выиграть:

• Не держать открытыми десятки вкладок браузера.
• Не запускать Docker Desktop, если не нужен.
• Не ставить тяжёлые антивирусы поверх Defender.
• Ограничить автозагрузку.


Что почти ничего не даст:

• Удаление стандартных приложений.
• Радикальное отключение служб.
• «Твики» реестра из случайных гайдов.

11. Идеальная FreeX Development Machine — от нуля до кода

1. Чистая установка Windows 11.
2. Проверка обновлений, установка всех драйверов.
3. Установка Winget/App Installer (если не стоит).
4. Через Winget: VS Code, Git, Node LTS, браузер.
5. Настройка Git (user, email, autocrlf, .gitignore).
6. Настройка SSH-ключа для GitHub.
7. Установка базовых расширений VS Code.
8. Клонирование репозитория FreeX.
9. Настройка Node-пакетов (npm/pnpm install).
10. Первый локальный запуск FreeX, проверка, что всё работает.
11. Настройка автозагрузки (минимум).
12. Создание отдельной папки FREEX-DEV для всех артефактов проекта (docs, scripts, backups).


---

Главные ошибки новичков

• Слишком агрессивно «оптимизируют» Windows, ломая обновления и безопасность.
• Ставят десятки расширений VS Code, превращая его в медленный комбайн.
• Коммитят секреты и ключи.
• Не делают резервные копии (ни кода, ни настроек).


---

Что обязательно проверить после настройки

• Работает ли Windows Update и Defender.
• Работает ли Git (push/pull/clone).
• Работает ли Node (версия, npm/pnpm).
• Открывается ли FreeX локально и на телефоне (через локальный сервер + туннель, если нужно).
• Нет ли лишних процессов в автозагрузке.


---

Что бы я сделал лично для FreeX

• Сохранил систему максимально стандартной, без радикальных твиков.
• Вложился в чистую структуру проекта, а не в «чистку» Windows.
• Настроил удобный терминал, Git, VS Code — чтобы каждый вход в систему был как вход в мастерскую, а не в лабиринт.


---

Самые удачные решения именно для FreeX

• Отдельная машина FREEX-DEV — это уже сильный шаг: контекст чистый, фокус высокий.
• Ставка на VS Code, Git, Node — это правильный стек для твоего проекта.
• Отказ от агрессивной «оптимизации» ради стабильности — это зрелое решение.


Если коротко: лучше пусть Windows будет немного «толще», но предсказуемой, чем «облегчённой», но хрупкой. Для FreeX важнее надёжность мышления, чем лишние 3% производительности.
