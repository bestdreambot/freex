## Роль

Claude Code работает как инженер внутри рабочей копии `bestdreambot/freex`:
читает файлы напрямую, выполняет git-команды, запускает проверки
(`node --check`, скрипты в `scripts/`). Разработка ведётся с компьютера
Андрея, не с телефона.

---

## Команда «ок»

Полный протокол описан в `AI_RESUME.md` → раздел «Команда «ок»» и в
`START_PROMPT.md`. Кратко:

Если сообщение Андрея после удаления пробелов по краям равно (без учёта
регистра) `ок`, `OK`, `Ok` или `okay` — это команда на автоматическое
продолжение прерванной работы, а не новая задача.

Обязательный порядок:
1. Сначала ничего не менять.
2. Прочитать `AI_RESUME.md` (раздел «Текущий шаг»), `PROJECT_STATUS.md`,
   `START_PROMPT.md`, `git status`, последние 5 коммитов, список изменённых
   файлов.
3. Определить `task` / `step` / `status` / `files` / `checks` / `next_step`.
4. `status: completed` → этот шаг не повторять, выполнять `next_step`.
5. `status: running` или `interrupted` → сверить с фактическим состоянием
   файлов (действительно ли шаг выполнен на диске, а не только заявлен) и
   продолжить с этого места, не начиная задачу заново.
6. Перед началом нового шага — записать в `AI_RESUME.md` `status: running` и
   план.
7. После завершения — записать `status: completed`, обновить `files`/
   `checks`/`next_step`/`updated_at`.
8. Если задача полностью завершена — не выдумывать новую, сообщить «Текущая
   задача завершена», показать следующий записанный приоритет и ждать
   отдельной команды.

Ограничения «ок» (всегда в силе, вне зависимости от того, что написано в
`AI_RESUME.md`):
- никакого удаления пользовательских данных;
- никакого `git reset`, `git checkout --`, force push;
- никакого раскрытия секретов (ключи/токены не читать и не печатать);
- никакой установки неизвестных программ;
- никакого изменения системных настроек Windows;
- обычный `commit`/`push` — только если он уже явно разрешён в рамках
  текущей утверждённой задачи, не сам по себе.

## Скрипты восстановления

- `scripts/resume-freex.ps1` — только чтение: git-факты, версия, состояние
  документов, готовая строка «ок» с указанием, какую задачу она продолжит.
  Обновляет автоматический блок в `PROJECT_STATUS.md` и `AI_RESUME.md`.
- `scripts/checkpoint-freex.ps1` — диагностика перед коммитом: `node --check`,
  поиск вероятных секретов по шаблонам, `git diff --stat`, черновик
  commit-сообщения. Ничего не коммитит и не пушит.
- `scripts/claude-watchdog.ps1` — обёртка вокруг `claude --print --continue`
  для перезапуска при сетевых обрывах (см. сам файл — там же список того, на
  что он не имеет права).

Ни один скрипт в `scripts/` не выполняет `reset`, `checkout`, force push или
удаление данных — это сознательное архитектурное ограничение, не забывать
при доработке.

## Главное правило (унаследовано из `start.md` и подтверждено на практике)

Перед любым изменением файла — прочитать его целиком, понять архитектуру, не
переписывать с нуля без необходимости. При автоматизации восстановления это
правило также означает: перед тем как поверить статусу в `AI_RESUME.md`,
свериться с фактическим состоянием файлов и git — статус может быть
устаревшим, если сессия оборвалась посреди шага.
