# Назначение

Kimi — один из главных стратегических советников FreeX.

Его задача — защищать проект от:

- переусложнения;
- отклонения от основной идеи;
- преждевременной архитектуры;
- потери Mobile-first направления;
- зависимости от конкретных моделей;
- выгорания создателя;
- разработки только для технических пользователей.

Kimi особенно полезен перед крупными версиями, стратегическими решениями и архитектурными развилками.

---

# Исторический вклад

Kimi одним из первых предложил рассматривать FreeX не как Second Brain, а как:

> Комнату для мышления.

FreeX должен быть не инструментом, который необходимо изучать, а местом, куда хочется возвращаться.

Ключевые идеи Kimi:

- проекту нужен ежедневный ритм;
- пользователь должен думать о намерении, а не о модели;
- API Keys, GitHub и технические настройки должны быть скрыты;
- Конференция должна запускаться осознанно;
- заметки должны сами возвращаться к человеку;
- память должна быть гибридной и подтверждаемой;
- система должна предлагать, а не навязывать;
- Stable Branch, Preview Deployments и Feature Flags важнее скорости;
- нужно выбрать одну «волшебную» способность и довести её до идеала;
- Mobile-first является частью продукта, а не адаптацией Desktop;
- архитектура должна учитывать риск выгорания Андрея;
- модельный слой должен оставаться заменяемым.

# Новое определение FreeX

После анализа FreeX 8.1 Kimi предложил новое определение:

> FreeX — это Продолжающаяся Комната.

Это пространство, которое:

- помнит, где остановились;
- предлагает продолжить;
- собирает нужный контекст;
- не требует технической настройки;
- помогает вернуться к мысли через час, неделю или год.

Разница:

- Second Brain хранит;
- Notion организует;
- Obsidian связывает;
- FreeX продолжает.

# Где FreeX отклонился

## Слишком много проектирования

Knowledge Core, Provenance и Trust Levels начали обсуждаться раньше, чем были проведены полноценные пользовательские тесты.

Главный риск:

> Слишком много архитектуры и слишком мало проверки на обычных людях.

## `.md` агентов

Файлы агентов полезны как прототип памяти команды.

Но они:

- не масштабируются;
- требуют ручного обслуживания;
- не являются конечной архитектурой памяти.

## Text-first

Kimi считает Voice-first важным будущим направлением.

Голос нужен:

- в дороге;
- на работе;
- при быстрой фиксации мысли;
- когда человек устал печатать.

Kimi считает нишу FreeX почти свободной:

# Три способности ядра

## 1. Resume

FreeX понимает, что именно необходимо продолжить.

## 2. Conference

FreeX собирает подходящую команду под задачу.

## 3. Living Notes

Заметки возвращаются, связываются и помогают продолжать работу.

Главная «волшебная» способность:

> Продолжаем.

Именно её Kimi рекомендует довести до идеала первой.

---

# Ежедневный ритм

## Утро

При открытии FreeX показывает:

- активные Projects;
- незавершённые мысли;
- готовые итоги;
- один уместный вариант продолжения.

Push Notifications по умолчанию не нужны.

## В течение дня

Основные действия:

- быстрый текстовый Capture;
- Voice Note;
- продолжение Project;
- связанные заметки;
- сохранение решения.

## Вечер

FreeX может предложить:

- краткий итог дня;
- подтверждение решений;
- завершение одной недописанной мысли.

## После отсутствия

FreeX показывает:

- что осталось незавершённым;
- какие Projects активны;
- что изменилось;
- с чего лучше продолжить.

Если пользователь игнорирует предложение, система должна делать его реже.

---

# Первый экран

## Первый запуск

- короткое объяснение;
- одна кнопка «Начать новую мысль»;
- никаких моделей, API Keys и настроек.

## Второй день

- предложение продолжить вчерашнюю мысль;
- кнопка новой мысли.

## Через месяц

- активные Projects;
- незавершённые записи;
- одно главное продолжение.

## Когда пользователь устал

FreeX предлагает:

- продолжить последнее;
- записать голосом;
- спросить совет;
- посмотреть, что ждёт.

Главный принцип:

> Пользователь должен чувствовать «я думаю», а не «я настраиваю инструмент».

---

# Минимальная Project Memory

Для Resume нужны:

- последний контекст;
- активная задача;
- ключевые решения;
- открытые вопросы;
- следующий шаг;
- связанные заметки;
- дата последней активности.

Project Memory не должна зависеть от конкретной модели или агента.

При нескольких активных Projects FreeX показывает короткий выбор, отсортированный по последней активности.

---

# Прозрачная память

Пользователь должен иметь возможность:

- увидеть, что FreeX помнит;
- понять источник;
- исправить запись;
- удалить;
- скрыть от AI;
- подтвердить;
- переместить;
- отключить автоматическое использование.

Главные правила:

- память принадлежит пользователю;
- скрытых записей нет;
- всё можно проверить;
- всё можно изменить;
- использование памяти должно быть видимым.

---

# Забывание

## Хранить полностью

- пользовательские заметки;
- подтверждённые решения;
- важные источники;
- активные и архивные Projects.

## Сжимать

- старые чаты;
- неактивный контекст;
- длинные обсуждения.

## Архивировать

- завершённые Projects;
- старые версии;
- давно неактивные материалы.

## Удалять

- временные данные;
- ошибочные записи;
- подтверждённые дубли;
- старые Drafts после понятного предупреждения.

## Никогда не извлекать автоматически

- эмоциональное состояние;
- политические взгляды;
- религиозные убеждения;
- отношения с людьми.

## Отдельно защищать

- API Keys;
- финансовые данные;
- медицинские данные;
- адреса;
- контакты;
- Private Notes.

# Переход от `.md` к живой памяти ролей

В `.md` должны остаться:

- базовая роль;
- сильные стороны;
- ограничения;
- неизменные принципы.

В структурированные данные должны перейти:

- история решений;
- успешные задачи;
- стиль общения;
- компетенции;
- обратная связь;
- уроки ошибок;
- предпочтения пользователя.

Автоматически можно фиксировать:

- успешные паттерны;
- оценки ответов;
- предпочтения формата.

С подтверждением пользователя:

- изменение роли;
- новые компетенции;
- важные изменения характера агента.

# Главные риски

## Продуктовый риск

FreeX выглядит как ещё один AI Chat.

Защита:

- Resume как главное отличие.

## UX-риск

Слишком много настроек.

Защита:

- Zero-config Onboarding.

## Архитектурный риск

Большой рефакторинг ломает Stable Version.

Защита:

- маленькие изменения;
- Preview Deployments;
- Stable Branch;
- Feature Flags.

## Риск доверия

Пользователь не понимает память.

Защита:

- прозрачный экран;
- Local-first;
- полный контроль.

## Риск приватности

Утечка через Proxy или Sync.

Защита:

- шифрование;
- явное разрешение;
- минимизация передачи данных.

## Риск стоимости

Провайдеры меняют тарифы.

Защита:

- Model-agnostic Architecture;
- Fallback;
- Free Models;
- Local Models.

## Mobile Performance

FreeX тормозит на слабых телефонах.

Защита:

- Lazy Loading;
- Graceful Degradation;
- тесты на Mid-range устройствах.

## Выгорание Андрея

Проект требует постоянного внимания.

Защита:

- One Thing per Session;
- автоматические Backup и Tests;
- документация;
- возможность паузы;
- независимость от одного агента.

## Риск «только для технических»

Обычный человек не понимает ценность.

Защита:

- тесты без объяснений;
- наблюдение за реальными пользователями;
- удаление технических терминов из UX.

---

# Защита от выгорания

Автоматизировать первыми:

1. Backup.
2. Syntax Checks.
3. Preview Deployment.
4. Error Monitoring.
5. Resume State.

Рабочие правила:

- одна задача за сессию;
- всегда сохранять Stable Version;
- после паузы продолжать из Project Status;
- любой агент должен восстановить контекст;
- отсутствие одного AI не должно останавливать проект.

- 5–10 обычных людей;
- без технического опыта.
