# Философия

FreeX должен стать не очередным AI-приложением.

Его цель —

стать личной операционной системой мышления.

Искусственный интеллект —

лишь один из инструментов.

Настоящее ядро системы:

- Memory
- Knowledge
- Context
- Trust

Главная задача FreeX —

не отвечать на вопросы,

а помогать человеку продолжать мысль.

---

# Основные принципы

## Knowledge важнее документов

Документы —

это контейнеры.

Настоящий объект системы —

Knowledge.

Markdown, заметки, чаты —

лишь способы хранения знаний.

Архитектура должна строиться вокруг Knowledge.

---

## Provenance First

Любой объект должен отвечать:

• Откуда появился?

• Кто создал?

• Когда создан?

• Почему существует?

• Кто изменял?

• Когда изменялся?

• Почему изменился?

• Можно ли ему доверять?

• Можно ли удалить?

• Кто владелец?

Происхождение —

обязательная часть архитектуры.

---

## Trust важнее Intelligence

Самая большая опасность —

не отсутствие интеллекта.

Самая большая опасность —

ложная уверенность.

FreeX всегда должен различать:

- remembered
- assumed
- verified
- canonical

Память без доверия превращается в шум.

---

## Спокойный интерфейс

Интерфейс должен снижать когнитивную нагрузку.

Не впечатлять.

Не развлекать.

А помогать думать.

Сложность должна раскрываться постепенно.

---

## Intent важнее модели

Пользователь думает задачами.

Не моделями.

Он должен выбирать:

• Подумать

• Написать

• Исследовать

• Сравнить

• Решить

• Придумать

FreeX сам выбирает подходящую модель.

При этом выбор должен быть объясним.

---

# Knowledge Core

Главный объект системы —

Knowledge.

Не документ.

Не чат.

Не заметка.

Любой объект Knowledge должен содержать минимум:

- id
- type
- title
- content
- created_at
- created_by
- source_refs
- confidence
- status
- scope
- review_at
- expires_at
- permissions
- provenance_chain

Дополнительно могут существовать:

- summary
- tags
- links
- attachments
- owners
- evidence_score
- alternatives
- counterarguments

---

# Основные типы Knowledge

- Fact
- Idea
- Hypothesis
- Decision
- Rule
- Task
- Project
- Event
- Source
- Entity
- Process

Новые типы должны появляться только при реальной необходимости.

Лучше использовать metadata,

чем создавать десятки классов знаний.

---

# Жизненный цикл Knowledge

Любое знание развивается.

Предлагаемый цикл:

Draft

↓

Reviewed

↓

Confirmed / Disputed

↓

Stale

↓

Archived / Deleted

Сохранение не означает истину.

Подтверждение —

отдельный процесс.

---

# Trust Levels

Предлагаемые уровни доверия:

1. Raw

2. Tentative

3. Supported

4. Verified

5. Canonical

Даже Canonical может быть пересмотрен.

---

# Multi-Agent Memory

Создавать Knowledge могут все агенты.

Подтверждать —

не все.

Рекомендуемая схема:

Agent

↓

Draft

↓

Validation

↓

Human или Validator Agent

↓

Canonical

Конфликтующие версии должны существовать параллельно,

пока не будет принято решение.

История никогда не уничтожается.

---

# Архитектура памяти

Perplexity предлагает три слоя.

## Evidence Layer

Неизменяемые источники.

Цитаты.

Импортированные документы.

История действий.

---

## Knowledge Layer

Нормализованные знания.

Связи.

Доверие.

Жизненный цикл.

Проверка.

---

## Reasoning Layer

Размышления.

Маршрутизация.

Планирование.

Рекомендации.

Принятие решений.

---

# Основные риски

Главные опасности:

• перепутать память с истиной;

• разрешить всем агентам одинаково изменять общую память;

• сделать слишком "умную" маршрутизацию без объяснений;

• перегрузить интерфейс;

• построить красивую архитектуру,

которой невозможно пользоваться.

# Основной принцип

FreeX должен помнить не только информацию.

Он должен помнить:

• откуда появилась информация;

• почему она существует;

• насколько ей можно доверять;

• когда её нужно пересмотреть;

• с чем она связана.

Knowledge должно быть:

- Explainable
- Verifiable
- Reviewable

Именно это является фундаментом настоящего второго мозга.
