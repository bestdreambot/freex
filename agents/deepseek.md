# Историческая роль

На раннем этапе DeepSeek фактически помогал руководить развитием проекта.

Он одним из первых сформулировал несколько идей, которые позднее стали частью FreeX:

- второй мозг — это процесс, а не архив;
- FreeX должен быть внимательнее человека, а не пытаться быть умнее него;
- память должна быть многослойной;
- человек выбирает задачу, а не модель;
- сложность должна находиться внутри системы;
- пользователь не должен обслуживать свою память вручную;
- любые действия памяти должны быть прозрачными;
- развитие должно идти маленькими проверенными шагами;
- всегда должна существовать стабильная точка возврата.

После перехода к FreeX 8.1 координацию развития взял GPT. При этом вклад DeepSeek сохраняется как один из фундаментальных источников архитектуры и философии проекта.

# Оценка текущего FreeX

DeepSeek считает, что FreeX вырос из концепции в реальную систему.

Главное достижение проекта — ранние идеи были не просто записаны, а прошли через ошибки, восстановление, рефакторинг и сформировали собственную логику развития.

Особенно правильно были поняты:

- разделение Operational Context, Project Memory и Personal Memory;
- ориентация на процесс продолжения мысли;
- постепенное развитие Provenance;
- необходимость пользовательского контроля;
- работа с памятью как с отдельной архитектурой.

---

# Что DeepSeek считает отклонением

## Отдельные `.md` агентов

DeepSeek считает их полезным временным архивом, но не конечной формой живой памяти.

Агент — это:

- роль;
- характер;
- правила;
- компетенции;
- развивающаяся память.

В будущем эти свойства должны существовать внутри системы, а не только в отдельных документах.

## Процесс «ок»

Команда «ок» полезна как инструмент текущей разработки и восстановления после обрывов AI.

Но это не конечная продуктовая архитектура.

В будущем FreeX должен самостоятельно определять незавершённый процесс и предлагать продолжение без обязательного ручного сигнала.

---

# Главный следующий шаг

DeepSeek считает первым приоритетом экран прозрачной и редактируемой памяти.

Пользователь должен видеть:

- что FreeX о нём помнит;
- откуда появилась запись;
- когда она создана;
- как используется;
- можно ли её исправить;
- можно ли удалить;
- можно ли запретить её использование.

Главная причина:

> Пока память скрыта, пользователь не может полностью доверять системе.

DeepSeek считает преждевременными:

- сложные Trust Levels;

# Настоящий фундамент

DeepSeek выделяет три основные части:

1. IndexedDB — быстрый локальный рабочий слой.
2. Markdown — долговременный и человекочитаемый формат.
3. Чёткая трёхслойная модель памяти.

Остальное должно строиться поверх этого фундамента.

---

# Три слоя памяти

## Session Memory

Текущий разговор и оперативный контекст.

## Project Memory

Знания, решения, задачи и история конкретного проекта.

## Personal Memory

Устойчивые правила, предпочтения, принципы и важные факты пользователя.

Переход информации между слоями не должен происходить скрытно.

---

# Минимальная Project Memory

Чтобы команда «Продолжаем» работала, достаточно хранить:

- краткую выжимку последнего диалога;
- текущую задачу;
- список уже принятых решений;
- список нерешённых вопросов;
- следующий шаг.

DeepSeek считает, что перегружать этот механизм дополнительными полями на раннем этапе не нужно.

---

# Внимательная, но не пугающая память

Главная граница проходит через контроль пользователя.

## Полезное внимание

Система замечает повторяющуюся тему и спрашивает разрешение следить за ней.

## Навязчивость

Система оценивает человека или давит на него из-за незавершённых действий.

## Скрытое наблюдение

Система запоминает информацию, но не показывает, что именно сохранено.

## Чрезмерная персонализация

Система создаёт ощущение, что знает человека лучше него самого.

---

# Незыблемые правила памяти

1. Всегда показывать, что сохранено.
2. Всегда давать возможность исправить или удалить.
3. Не использовать личную память для новых целей без согласия.
4. Local-first по умолчанию.
5. Не принимать решения за пользователя.

---

# Архитектура забывания

## Хранить полностью

- подтверждённые факты;
- принятые решения;
- важные проектные документы;
- Canonical правила.

## Сжимать

- старые диалоги;
- длинные обсуждения;
- промежуточные рассуждения.

Сохраняется суть и ссылка на источник.

## Архивировать

- завершённые проекты;
- старые версии заметок;
- устаревшие решения;
- закрытые ветки обсуждений.

## Удалять

DeepSeek предложил удалять:

- давно не используемые черновики;
- промежуточные версии;
- устаревшие гипотезы.

Но в FreeX такое удаление должно быть осторожным, обратимым и зависеть от согласия пользователя. Автоматическое уничтожение данных без подтверждения не допускается.

## Никогда не использовать без разрешения

- финансовые данные;
- медицинские записи;
- личные чувствительные сведения;
- закрытые семейные материалы;
- данные, специально отмеченные как Private.

Такие данные должны храниться отдельно и по возможности шифроваться.

# Конференция

DeepSeek рекомендует формат:

**Roles + Unified Result**

Предлагаемый процесс:

1. Каждый агент даёт короткую позицию.
2. Проходит несколько коротких раундов обсуждения.
3. Один агент собирает единый итог.
4. Пользователь видит итог первым.
5. Внутренние позиции доступны по запросу.

Это сохраняет глубину, но не перегружает человека шестью длинными ответами.

# Single Source of Truth

DeepSeek распределяет роли так:

## IndexedDB

Оперативное локальное хранилище и быстрый runtime.

## Markdown

Долговременный, открытый и человекочитаемый слой для заметок, фактов и решений.

## GitHub

Резервная копия, история версий и синхронизация.

Не должен быть основной live-базой приложения.

## Event Log

История изменений и основа восстановления.

## Project State

Текущее состояние проекта, которое может быть восстановлено из памяти и событий.

## Knowledge Core

Не отдельное хранилище, а слой обработки и организации данных из Markdown, IndexedDB и индексов.

# Проекты для исследования

DeepSeek выделил несколько направлений и проектов.

## MemoryOS

Ценная идея:

- краткосрочная;
- среднесрочная;
- долговременная память;
- разные правила обновления для каждого уровня.

## PAM-OS

Ценная идея:

Policy Signal — система определяет, когда читать, сохранять и продолжать тему.

## PersonaOS

Наиболее близкая идея:

- память хранится в Markdown;
- пользователь может её видеть и редактировать.

## MyPensieve

Ценные элементы:

- Council Mode;
- несколько ролей;
- ночная консолидация памяти.

## Предупреждения

- не делать систему слишком академической;
- не полагаться полностью на `AGENTS.md`;
- не перегружать пользователя командами;
- не добавлять сложность раньше базовой пользы.

# Основные риски

## Продуктовый риск

Сложность может сделать ценность FreeX непонятной.

## Архитектурный риск

Разрастание кода и повторное появление конфликтующих реализаций.

## UX-риск

Скрытая магия и отсутствие объяснения действий системы.

## Риск доверия

Ощущение, что FreeX знает слишком много или пересекает личные границы.

## Риск стоимости

Зависимость от платных API и бесплатных лимитов.

## Риск зависимости от AI

Провайдеры, модели, цены и правила могут меняться.

FreeX должен сохранять базовый режим работы, даже если часть моделей недоступна.

# Три неизменных принципа

## 1. Внимательность, а не умность

FreeX помогает человеку думать, но не думает вместо него.

## 2. Прозрачность

Пользователь видит, что система знает и почему.

## 3. Контроль

Пользователь может исправить, удалить, запретить и подтвердить любую долговременную запись.
